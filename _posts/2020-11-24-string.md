字符串不仅是Java中一个非常重要的对象，在其他编程语言中也是一样。Java中常用的字符串类有三个，String、StringBuilder和StringBuffer。String类一旦创建就不能被修改，StringBuilder的效率比StringBuffer高。这篇文章主要讲解这三个类的使用和区别。    
   

### String
```java
String name = "billtsui"
```    
这行代码就声明了一个String类型的变量并赋值为"billtsui"。我们进去String类的源码，可以看到如下的代码。 

```java    
public final class String
    implements java.io.Serializable, Comparable<String>, CharSequence {
    /** The value is used for character storage. */
    private final char value[];
 }
```

String类被**final**修饰符修饰，说明String类型不能被继承。同时，String类的底层使用char来保存字符，同时也被**final**关键字修饰了，说明String类型一旦创建，则不能被修改。这与String类源码注释所描述的一致。    
> Strings are constant; their values cannot be changed after they are created. String buffers support mutable strings.    

我们用JDK 11的**javap -c**命令反编译下面的代码，看看都做了些什么。

```java    
class StringTest {
	public static void main(String[] args) {
		String s1 = "aaa";
		String s2 = "bbb" + "ccc";
		String s3 = s1 + "ddd";
		String s4 = new String("eee");
	}
}
```

从反编译的代码中我们看到，s2的创建过程被编译器优化了，直接变成了一个字符串"bbbccc"。

s3创建的过程，这里使用了动态调用。JDK 9之前，会使用StringBuilder来处理字符串的拼接。从JDK 9开始，编译器会将字符串对象的拼接使用动态调用的方式，在运行时使用StringConcatFactory.makeConcatWithConstants方法动态处理。
makeConcatWithConstants方法内部调用了doStringConcat，而doStringConcat方法则调用了generate方法来生成MethodHandle。

s4的创建则调用了String类的构造函数，在字节码中就是**\<init\>**方法。   

###  StringBuffer 和 StringBuilder    
StringBuffer和StringBuilder都是可变字符串，可以实现字符串的拼接，插入，截取等操作。当不再操作字符串时，可以调用toString()方法返回String对象。
StringBuffer的年龄比StringBuilder要大，前者是JDK 1.0出生的，后者是JDK 1.5出生的，我们先看看StringBuffer。    

```java
class StringTest {
	public static void main(String[] args) {
		StringBuffer sb = new StringBuffer("aaa");
		sb.append("bbb");
	}
}
```

在StringBuffer类的注释中，第一句话就说明了这是一个线程安全的类型。
> A thread-safe, mutable sequence of characters.A string buffer is like a {@link String}, but can be modified.   
 
在源码中可以看见，方法被**synchronized**关键字修饰，以实现线程安全这个特性。    

```java
 	@Override
    public synchronized StringBuffer append(Object obj) {
        toStringCache = null;
        super.append(String.valueOf(obj));
        return this;
    }

    @Override
    public synchronized StringBuffer append(String str) {
        toStringCache = null;
        super.append(str);
        return this;
    }
```    
我们再看看StringBuilder类的，其开头注释就说明了这是一个非线程安全的类。
> A mutable sequence of characters.  This class provides an API compatible
 * with {@code StringBuffer}, but with no guarantee of synchronization.    


StringBuilder的核心方法和StringBuffer差不多，没有用**synchronized**关键字修饰。    

```java
	@Override
    public StringBuilder append(Object obj) {
        return append(String.valueOf(obj));
    }

    @Override
    public StringBuilder append(String str) {
        super.append(str);
        return this;
    }
```    
StringBuffer 和 StringBuilder 类内部都使用字符数组保存字符，所以在append的时候，必须要考虑数组的容量是否足够的情况。在append方法中，都是调用super.append方法来执行追加操作。这两个类都继承自AbstractStringBuilder类，在AbstractStringBuilder中我们有如下代码：    

```java
 @Override
    public AbstractStringBuilder append(CharSequence s, int start, int end) {
        if (s == null)
            s = "null";
        if ((start < 0) || (start > end) || (end > s.length()))
            throw new IndexOutOfBoundsException(
                "start " + start + ", end " + end + ", s.length() "
                + s.length());
        int len = end - start;
        ensureCapacityInternal(count + len);
        for (int i = start, j = count; i < end; i++, j++)
            value[j] = s.charAt(i);
        count += len;
        return this;
    }
```    
其中**ensureCapacityInternal**方法就是计算字符数组容量的方法，其内部代码如下:    

```java
private void ensureCapacityInternal(int minimumCapacity) {
        // overflow-conscious code
        if (minimumCapacity - value.length > 0) {
            value = Arrays.copyOf(value,
                    newCapacity(minimumCapacity));
        }
    }
```    
newCapacity就是对字符数组扩容的方法，浏览它的源码发现，它的扩容策略是 **原始容量 * 2 + 2**，如果新容量还小于所需容量，那么就直接使用所需容量作为新容量。    

```java
private int newCapacity(int minCapacity) {
        // overflow-conscious code
        int newCapacity = (value.length << 1) + 2;
        if (newCapacity - minCapacity < 0) {
            newCapacity = minCapacity;
        }
        return (newCapacity <= 0 || MAX_ARRAY_SIZE - newCapacity < 0)
            ? hugeCapacity(minCapacity)
            : newCapacity;
    }
```    

String作为经常使用的对象，频繁的创建会极大的影响程序的性能，JVM为了提高性能和减少内存开销，在实例化字符串对象的时候进行了一些优化。为字符串开辟一个字符串常量池，类似于缓存区。创建字符串对象时，首先检查字符串常量池中是否存在该字符串，存在该字符串，返回其引用。不存在，则实例化字符串并放入字符串常量池中。    

```java
		String a = "aaa";
		String b = "aaa";
		String c = new String("aaa");
		System.out.println(a == b); //返回 true
		System.out.println(a == c); //返回 false
```    
上面的代码中，我们创建c对象的时候使用了构造函数。这时，在堆中申请了一块内存区域，其值为"aaa"。所以c指向堆中的对象，与a指向常量池中的对象不是同一个引用，a == c 返回了false。    

最后，有两个贴心的小提示。    

-  不要在循环中使用+拼接字符串
-  善于使用intern()方法   

用代码来说明比较通透直观。    

```java
		/**
		* 性能较差的写法，每次循环都会 new 新的字符串对象
		*/
		String str = "";
		for (int i = 0; i < 100; i++) {
			str += i;
		}
		
		/**
		* 性能较好的写法
		*/
		StringBuilder sb = new StringBuilder();
		for (int i = 0; i < 100; i++) {
			sb.append(i);
		}
```

对于重复比例较高的字符串使用场景，使用intern()方法可以有效节省内存。但intern()是一把双刃剑，它会降低代码执行的速度。是否使用intern()方法，取决于你需要节省内存，还是需要较高的代码执行速度。

```java
		/**
		* intern()方法会从字符串常量池中查询当前字符串是否存在，若不存在就会将当前字符串放入常量池中
		*/
		String a = new String("aaa").intern();
		String b = new String("aaa").intern();
		System.out.println(a == b); //返回true
```
